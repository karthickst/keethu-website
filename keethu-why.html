Why Keethu Exists

Modern financial systems, game servers, IoT platforms, and real-time dashboards all share one brutal requirement: many clients need to receive the same data stream simultaneously, with predictable low latency. A stock feed must reach 50,000 trading terminals within milliseconds of the market moving. A gaming server must push position updates to 10,000 players every 16ms.

The naive solution — one thread per connection, each thread reading from a shared queue — collapses at scale. At 10,000 connections you have 10,000 threads. Each thread consumes ~8MB of stack by default on Linux. That's 80GB of stack memory before you've processed a single message. Context-switching 10,000 threads costs hundreds of milliseconds per cycle.

Keethu is built to answer: how do you deliver messages to 100,000 concurrent subscribers with ≤10ms p99 latency on a single machine? Every architectural decision in Keethu exists to answer that question. This guide explains the physics behind those decisions.

CHAPTER 1

CPU Internals Novice

A modern CPU does not simply "run instructions one by one." It uses a deep pipeline to execute many instructions simultaneously, and it aggressively reorders, speculates, and caches to keep all execution units busy.

The Pipeline

Each instruction passes through stages: Fetch → Decode → Rename → Dispatch → Execute → Commit. Intel's Skylake CPU has a 14-19 stage pipeline. The pipeline is wide: a modern core can retire up to 4 instructions per clock cycle (superscalar execution).

Cycle: 1 2 3 4 5 6 7 8 9 Instr1: F - D - E - W Instr2: F - D - E - W Instr3: F - D - E - W Instr4: F - D - E - W F=Fetch D=Decode E=Execute W=Writeback All four instructions complete by cycle 7 instead of cycle 16

Branch Prediction

When the CPU hits an if statement, it doesn't wait — it predicts which branch will be taken and starts executing speculatively. Modern CPUs predict correctly ~99% of the time. A misprediction flushes the pipeline (~15-20 cycles wasted). This is why tight loops with unpredictable branches are slow.

Keethu Connection

The hot path in Keethu's fan-out loop iterates subscribers. If the subscription list is always non-empty (the common case), the branch predictor stays warm. We avoid conditional checks inside the tight loop for this reason.

Cache Hierarchy Novice

RAM is slow. The CPU cannot wait 70+ nanoseconds for every data fetch. So it keeps frequently-used data in progressively smaller, faster caches closer to the execution units.

Cache Lines

CPUs don't fetch individual bytes — they fetch a cache line of 64 bytes at a time. If you access one byte at address X, the CPU loads bytes X through X+63 into cache. This makes sequential memory access (arrays) much faster than random access (linked lists), because the next element is already in cache.

// Linked list: each node is at a random address → cache miss per node
struct Node { val: u64, next: *mut Node }

// Vec: elements are contiguous → next element already in cache line
let v: Vec<u64> = vec![1, 2, 3, ...]; // blazing fast iteration
Why This Matters for Keethu

Keethu's subscription list for each topic is a Vec<SubEntry>, not a linked list. During fan-out, iterating 1,000 subscribers in a Vec is far faster than traversing a linked list — the CPU prefetcher can stream cache lines ahead of the iterator.

NUMA — Non-Uniform Memory Access Intermediate

In a multi-socket server, each CPU socket has its own local DDR channels. Accessing memory local to your socket is fast (~80 ns). Accessing memory on the other socket crosses the interconnect (Intel UPI or AMD Infinity Fabric) and costs ~120–140 ns — 50–75% slower.

Socket 0 Socket 1 ┌──────────────────┐ UPI ┌──────────────────┐ │ Cores 0-31 │ ◄──────► │ Cores 32-63 │ │ L3 Cache 64MB │ ~40ns │ L3 Cache 64MB │ │ DDR5 256GB │ │ DDR5 256GB │ └──────────────────┘ └──────────────────┘ Local access: ~80ns Cross-socket: ~140ns

Keethu's recommended machine spec calls for a single-socket AMD EPYC 9374F precisely to avoid NUMA complexity. When you must use dual-socket, you pin the Tokio runtime to socket 0's cores and allocate all memory with numactl --membind=0.

SIMD — Single Instruction, Multiple Data Advanced

SIMD instructions operate on vectors of data simultaneously. AVX2 processes 256 bits (32 bytes) per instruction. AVX-512 processes 512 bits (64 bytes). Applied to JSON parsing — which Keethu uses for frame headers — SIMD can scan for the closing } brace and validate UTF-8 in 32-byte chunks instead of byte-by-byte, achieving 3-5× throughput improvement.

// Scalar: check one byte at a time
for byte in input { if byte == b'}' { /* found */ } }

// SIMD (via simd-json crate): compare 32 bytes simultaneously
// _mm256_cmpeq_epi8(chunk, splat(b'}')) → bitmask of matches
// 32× fewer iterations for the same input

📖 FAQ — Computer Architecture

Why can't we just add more CPU cores to fix latency?

More cores help throughput (more work done per second) but don't inherently reduce latency (time for one piece of work to complete). If a single message delivery requires acquiring a lock, the lock contention can make adding cores make things worse — each core spins burning CPU while waiting. Keethu uses lock-free data structures precisely so that more cores mean more fan-out parallelism with no contention penalty.

What is a cache miss and how costly is it really?

A cache miss occurs when data needed by the CPU isn't in L1/L2/L3 cache, forcing a fetch from RAM. An L1 hit takes ~1 ns. An L3 miss (RAM access) takes ~70 ns — 70× slower. In a tight loop processing 1M messages/second, each message that causes an L3 miss costs 70ns × 1M = 70ms of aggregate stall time per second. Keeping hot data structures (subscription lists, sequence counters) within L3 cache is critical for Keethu's latency targets.

What is false sharing and how does it hurt performance?

False sharing occurs when two CPU cores modify different variables that happen to reside on the same 64-byte cache line. Even though the variables are logically independent, the cache coherence protocol forces cores to coordinate on every write, serializing what should be parallel work. Fix: pad structs so hot independent fields live on different cache lines.

// BAD: counter_a and counter_b share a cache line
struct Bad { counter_a: AtomicU64, counter_b: AtomicU64 }

// GOOD: each on its own cache line
struct Good {
  counter_a: AtomicU64,
  _pad: [u8; 56], // fill to 64 bytes
  counter_b: AtomicU64,
}
What is out-of-order execution?

Modern CPUs don't execute instructions in the order they're written. They analyze the dependency graph between instructions and execute independent ones in parallel. If instruction 3 doesn't depend on the result of instruction 2, the CPU can start executing instruction 3 while instruction 2 is still waiting for a memory access. This is invisible to the programmer — the CPU guarantees correct results — but it means that code with fewer data dependencies is faster.

CHAPTER 2

Operating Systems — Processes & Threads Novice

An operating system manages hardware resources (CPU, memory, devices) and provides abstractions that programs use without knowing the hardware details. Understanding the OS is essential because Keethu lives at the boundary between user code and OS kernel.

Process vs Thread

ConceptProcessThread
Memory spaceOwn virtual address spaceShared with all threads in process

A context switch happens when the OS scheduler decides to pause thread A and resume thread B. It saves all of A's CPU registers to memory, loads B's registers, and (for processes) flushes the TLB (Translation Lookaside Buffer — the cache for virtual→physical memory translations). This costs time and pollutes caches.

The Thread-per-Connection Trap

A classic server spawns one thread per client connection. At 10,000 connections: 10,000 × 8MB stack = 80GB RAM just for stacks. The scheduler spends more time context-switching than doing real work. This is the C10K problem, described by Dan Kegel in 1999. Keethu uses async tasks — which have ~KB-sized state rather than MB-sized stacks — to handle 100K connections with a small fixed thread pool.

System Calls Novice

Programs cannot directly access hardware. They must ask the OS via system calls. Each syscall involves:

  1. Saving user registers
  2. Switching CPU privilege level from Ring 3 (user) to Ring 0 (kernel)
  3. Executing kernel code
  4. Switching back to Ring 3 and restoring registers

A syscall round-trip costs ~100–300 ns on a modern CPU. That sounds small, but at 1M messages/second, calling write() once per message wastes 100–300 ms/second on syscall overhead alone — 10–30% of your latency budget gone.

Keethu's approach

Keethu writes to TCP sockets via Tokio's async I/O, which batches multiple writes into a single write() syscall wherever possible using writev() (scatter-gather I/O). The codec layer assembles frames into a BytesMut buffer before calling into the kernel.

I/O Models — The Road to epoll Intermediate

There are four fundamental ways a program can perform I/O. Understanding the evolution explains why Keethu uses async I/O.

1. Blocking I/O

The thread calls read(fd) and sleeps until data arrives. Simple, but the thread is useless while blocked. Requires one thread per socket → the C10K disaster.

2. Non-Blocking I/O

Set the socket to O_NONBLOCK. read(fd) returns immediately with EAGAIN if no data. The thread can do other work. But now you must poll all sockets repeatedly to check — if you have 10,000 sockets, that's 10,000 syscalls per polling cycle. At 10ms polling interval: 10,000 syscalls × 100 ns = 1 ms wasted on polling overhead each cycle.

3. select / poll

Give the kernel a list of file descriptors. The kernel wakes you when any of them are ready. select is limited to 1,024 FDs. poll is unlimited but passes the entire FD list to the kernel on each call — O(n) per call. Still O(n) overhead as connections grow.

4. epoll — O(1) I/O Multiplexing Intermediate

epoll (Linux ≥2.5.44) solves the O(n) problem. You register interest in FDs once. The kernel maintains a ready list. When an FD becomes readable/writable, the kernel adds it to the ready list. epoll_wait() returns only the ready FDs — O(1) regardless of total connection count.

Traditional poll: on each call, scan all FDs poll([fd1, fd2, ..., fd100000], timeout) ← copies 100K FDs to kernel every call epoll: register once, get notifications epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &event) ← once per connection epoll_wait(epfd, events, maxevents, timeout) ← returns ONLY ready FDs

Edge-Triggered vs Level-Triggered

epoll has two modes. Level-triggered (LT): epoll_wait keeps returning an FD as long as it's ready. Edge-triggered (ET): epoll_wait fires only when the state changes (new data arrives). ET is more efficient but requires you to read all available data in one shot (use a loop until EAGAIN).

Tokio uses epoll in edge-triggered mode internally. This is one reason Tokio can handle 100K connections on a single thread efficiently.

Virtual Memory & Huge Pages Intermediate

The OS gives each process the illusion of a flat 64-bit address space. Physical RAM is sliced into 4KB pages. The CPU's MMU (Memory Management Unit) translates virtual addresses to physical addresses using a page table. The TLB caches recent translations.

With 4KB pages: to address 128GB of RAM, you need 128GB / 4KB = 32 million page table entries. A TLB miss requires walking this tree — 3–5 memory accesses, 150–400 ns. With huge pages (2MB), you need only 65,536 entries — 512× fewer TLB misses.

# Enable transparent huge pages for Keethu process
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
# In Rust: hint the allocator to use huge pages for large buffers
madvise(ptr, size, MADV_HUGEPAGE)

📖 FAQ — Operating Systems

How does async/await avoid the thread-per-connection problem?

An async task is implemented as a state machine compiled by the Rust compiler. Instead of a thread (8MB stack), an async task stores only the variables that survive across await points — typically a few hundred bytes. 100,000 async tasks cost ~10MB of state, not 800GB of stack. The Tokio runtime runs these tasks on a small pool of OS threads (typically one per CPU core), using epoll to know which tasks have I/O ready.

What is a file descriptor and why is the default limit only 1,024?

A file descriptor (FD) is a small integer the kernel assigns to represent an open file, socket, pipe, or device. The default per-process limit (ulimit -n) is 1,024 for historical reasons (it fit in a 32-bit bitmask used by the original select() call). For Keethu with 100K connections, you must set ulimit -n 1048576 and fs.file-max = 2000000 in sysctl. Each TCP connection consumes two FDs (one for the socket, one internally).

What is the difference between a kernel thread and a green thread?

A kernel thread is managed by the OS scheduler — it has an 8MB stack, appears in ps, and is preemptively scheduled. A green thread (or async task) is managed by a user-space runtime (like Tokio) — it has a tiny state machine instead of a stack, is cooperatively scheduled (it yields at await points), and the OS doesn't know it exists. Tokio uses M:N threading: M async tasks multiplexed onto N kernel threads (N ≈ CPU core count).

Why does mmap help for large data, and when should you avoid it?

mmap maps a file directly into your process's virtual address space. Reads/writes go directly to the page cache — no read()/write() syscall overhead. This works great for large sequential reads (SOW replay in Keethu). However, mmap access can trigger page faults (OS loading a 4KB page from disk) at unpredictable times, causing latency spikes. For latency-sensitive paths, pre-fault pages with madvise(MADV_WILLNEED).

CHAPTER 3

Networking — TCP Internals Novice

Keethu communicates over TCP. Understanding TCP's machinery explains every sysctl we tune and every latency spike we work around.

The Three-Way Handshake

Client Server │──── SYN ────────────►│ Client sends: "I want to connect, seq=X" │◄─── SYN-ACK ─────────│ Server: "OK, seq=Y, ack=X+1" │──── ACK ────────────►│ Client: "ack=Y+1" │ │ │ Connection ESTABLISHED│ │ Round-trip time: 1 RTT│

A TCP connection requires one round-trip before any data flows. On a LAN, RTT ≈ 0.1ms. Keethu clients connect once and maintain persistent connections — this one-time cost is amortized over millions of messages.

Nagle's Algorithm — The Silent Latency Killer

Nagle's algorithm (enabled by default) coalesces small TCP writes. If you call write() with 10 bytes, Nagle may wait up to 200ms for more data to accumulate before actually sending the packet. This devastates interactive, low-latency protocols.

// Keethu disables Nagle immediately on accept
stream.set_nodelay(true)?; // TCP_NODELAY: send immediately, don't wait

Socket Buffers Intermediate

Every TCP socket has two kernel buffers: a send buffer (data waiting to be transmitted) and a receive buffer (data received but not yet read by the application). These buffers live in kernel memory.

Application Kernel Network │ ┌──────────────┐ │──write(data)──► │ Send Buffer │──TCP segments──► wire │ │ (128KB) │ │ └──────────────┘ │ ┌──────────────┐ │◄──read(data)── │ Recv Buffer │◄─TCP segments─── wire │ │ (128KB) │ │ └──────────────┘

If the send buffer is full (slow consumer, network congestion), write() blocks (or returns EAGAIN for non-blocking). This is back-pressure — the OS is telling you to slow down. Keethu's per-connection mpsc channel adds another layer of back-pressure before data even reaches the socket.

Zero-Copy Networking Advanced

In the traditional path, sending data copies it four times:

  1. Application buffer → kernel page cache (read())
  2. Kernel page cache → socket send buffer (write())
  3. Socket send buffer → NIC DMA buffer
  4. NIC DMA buffer → wire

Zero-copy techniques eliminate intermediate copies. sendfile() lets the kernel copy from page cache directly to NIC — bypassing user space entirely. For Keethu's in-memory pub/sub, reference counting is the zero-copy technique: the payload Bytes object is shared among all subscribers via an Arc, no copying.

// Keethu publish: payload is cloned zero-copy via Bytes reference counting
pub fn publish(&self, topic: &str, payload: Bytes, publisher: &str) {
    for sub in subs.iter() {
        let frame = delivery_frame(&sub.sub_id, topic, &payload, ...);
        // payload.clone() is O(1) — just increments a reference count
        // no data is copied until the last Arc is dropped
        sub.tx.try_send(frame);
    }
}

TIME_WAIT Explained

When a TCP connection closes, the closing side enters TIME_WAIT for 2×MSL (~60–120 seconds) to ensure the final ACK was received. With 100K connections/hour connecting and disconnecting, you could accumulate thousands of TIME_WAIT sockets. tcp_tw_reuse=1 allows reusing these for new outbound connections after 1 second.

📖 FAQ — Networking

Why does TCP have a "slow start" and how does it affect pub/sub?

TCP's slow start algorithm begins each connection (and after idle) with a small congestion window (10 packets ≈ 14KB). It doubles every RTT until it hits the congestion threshold or packet loss. For a client subscribing and then immediately receiving a burst of messages, the first few hundred milliseconds may be rate-limited by slow start. Persistent connections (keep-alive) avoid this — Keethu clients connect once and stay connected, so slow start only affects the initial handshake, not message delivery.

What is RSS (Receive Side Scaling) and why does Keethu enable it?

By default, a NIC handles all incoming packets on a single CPU core — a bottleneck at high packet rates. RSS distributes incoming connections across multiple CPU cores based on a hash of (src IP, src port, dst IP, dst port). Each core processes its own subset of connections independently, with no lock contention. Keethu's machine spec recommends configuring RSS queues equal to the number of CPU cores used by Tokio: ethtool -L eth0 combined $(nproc).

What is the difference between TCP_NODELAY and TCP_CORK?

TCP_NODELAY disables Nagle's algorithm — each write is sent immediately as its own packet. TCP_CORK is the opposite: hold all data until either the buffer is full (MSS bytes) or the cork is removed. Use TCP_CORK when you're assembling a response from multiple write() calls and want them sent as one packet. Keethu uses TCP_NODELAY on all client sockets; it assembles complete frames before writing, so Nagle-style coalescing is unnecessary and harmful.

What happens when a subscriber is too slow to consume messages?

The chain is: Keethu's mpsc channel fills up → try_send() returns Err(Full) → Keethu logs "slow consumer" and drops the message. The subscriber's TCP receive buffer may also fill up → the OS stops ACKing → TCP's receive window shrinks to zero → the sender (Keethu) cannot push more data into the socket → back-pressure propagates all the way up. Keethu's design choice is to drop messages for slow consumers rather than let one slow subscriber stall all others.

CHAPTER 4

Performance Engineering Intermediate

Latency vs Throughput — Not the Same Thing

Latency is how long one operation takes end-to-end — measured in nanoseconds to milliseconds. Throughput is how many operations the system completes per second. These are related, but they pull in opposite directions: the technique that maximises one often hurts the other.

Throughput Optimisation

The goal is to move as many bytes as possible per second. The main cost is syscall overhead — each write() into the kernel takes ~100–300 ns regardless of payload size.

Accumulate 64 KB in a buffer
→ one write() syscall
→ kernel sends one large TCP segment
→ high msg/sec ✓ higher latency ✗

Trade-off: data sits in the buffer waiting for it to fill up. A message may wait milliseconds before it's flushed — unacceptable for real-time delivery.

Latency Optimisation

The goal is to deliver each message as fast as possible. Write to the socket the moment a frame is ready — never wait.

Frame ready → write() immediately
→ kernel sends small TCP segment
→ subscriber receives it right away
→ low latency ✓ fewer msg/sec ✗

Trade-off: one syscall per message. At 1 M messages/sec that's 100–300 ms/sec wasted on syscall overhead alone — 10–30% of your CPU budget gone.

Keethu's Solution — Async Pipeline

Keethu must achieve both simultaneously: ≤10 ms p99 latency and 1 M+ messages/second throughput. The key insight is to never block at the application layer, but let the OS batch at the kernel layer.

Each subscriber has a bounded async mpsc channel. The fan-out loop pushes frames into channels with a non-blocking try_send() (~10 ns each). A separate async write task drains the channel and calls writev() — scatter-gather I/O that lets the kernel coalesce multiple frames into one syscall automatically, without adding artificial delay. The frames leave as fast as the socket allows, not as fast as the buffer fills.

Little's Law

A fundamental theorem of queuing theory: L = λW. The average number of items in a system (L) equals the arrival rate (λ) times the average time each item spends in the system (W). If 100,000 messages/second arrive and each takes 10ms to process: L = 100,000 × 0.01 = 1,000 messages in-flight at any time. Your in-flight buffer must be sized accordingly.

Percentiles — Why p99 Matters More Than Mean Intermediate

Average latency is nearly meaningless for real systems. A system that handles 99% of requests in 1ms but 1% in 10 seconds has an "average" of ~100ms — but users experience the 10-second tail. For Keethu with 100K subscribers, 1% of 100K = 1,000 subscribers experiencing bad latency simultaneously.

Latency spikes — the outliers that inflate your p99 — each have a distinct root cause. Here is what each one means in plain terms, and how Keethu addresses it.

Eliminated by Rust Garbage Collection Pauses

In Java or Go, a GC thread periodically stops every other thread to find and free unused memory — like stopping a highway to sweep it. For 10–200 ms the server does nothing. Every subscriber waiting for a message just… waits. Rust has no GC: memory is freed deterministically the instant its owner goes out of scope. No sweeper, no pause, ever.

Mitigated by Tokio OS Scheduler Jitter

The OS scheduler gives each thread a time slice — typically 1–4 ms — then forcibly pauses it to let another thread run, even if the first thread had urgent work. Imagine a chef being told to stop mid-sentence every few seconds so the dishwasher can speak. Tokio keeps just one thread per CPU core, so the OS almost never needs to preempt them. Fewer threads = far less scheduler jitter.

Mitigated by TCP_NODELAY TCP Retransmissions

If a network packet is lost (a router drops it under load), TCP waits for a timeout — typically 200 ms — before resending. The subscriber receives nothing during that window. Keethu runs on a LAN where packet loss is rare (<0.01%), and persistent keep-alive connections avoid the slow-start retransmission burst that cold connections suffer. For the rare retransmission event, Keethu's mpsc buffer absorbs the gap so other subscribers aren't affected.

Mitigated by DashMap + Atomics Lock Contention

A mutex is like a single-stall bathroom: only one person at a time. When 32 CPU cores all try to publish at once through a Mutex<HashMap>, 31 cores are just standing in line. Each wait can cost 1–5 µs — at a million operations per second that's your entire latency budget gone. Keethu uses DashMap (128 independent shards) and atomic operations so cores almost never share a lock.

Mitigated by madvise / huge pages Page Faults

Virtual memory is divided into 4 KB pages. The first time your code touches a new page — even one you just allocated — the OS must wire it to real RAM, a process called a page fault. It costs 1–10 µs and happens at an unpredictable moment. Keethu pre-faults its large buffers at startup with madvise(MADV_WILLNEED) and uses huge pages (2 MB) to need 512× fewer TLB lookups, keeping the hot path page-fault-free.

Mitigated by single-socket spec NUMA Remote Accesses

In a two-CPU server, each processor has its own RAM bank. If CPU 0 needs data that was allocated in CPU 1's RAM, it must cross the interconnect — adding ~60 ns compared to a local read. That's 60× slower than an L1 cache hit. Keethu's recommended machine is a single-socket AMD EPYC, so all cores share one memory pool. If dual-socket is unavoidable, numactl --membind=0 forces all allocations to socket 0.

Mitigated by CPU pinning / cooling Thermal Throttling

When a CPU core overheats it automatically slows its clock — a 3.8 GHz core may drop to 2.4 GHz or lower. Instructions that took 0.26 ns now take 0.42 ns; over millions of operations the cumulative delay is measurable. Keethu's ops guide recommends dedicated server hardware with adequate cooling, disabling "Turbo Boost" (which causes clock-speed swings), and monitoring core temperatures with sensors before load-testing.

Lock-Free Programming Advanced

A mutex (mutual exclusion lock) allows only one thread to access shared data at a time. When thread A holds the lock and thread B wants it, thread B is suspended by the OS. When A releases, the OS must wake B — this context switch costs ~1–5µs. At 1M operations/second, lock contention can cost 10–50% of your entire time budget.

Atomic Operations

CPU instructions like Compare-And-Swap (CAS) and Fetch-And-Add (FAA) operate on a single memory location atomically without a mutex. They use CPU-level locking (cache line lock) which takes ~5–20 ns instead of ~1–5µs.

// Keethu's global sequence counter — no mutex needed
pub seq: AtomicU64,

// fetch_add is a single atomic CPU instruction (LOCK XADD on x86)
let seq = self.seq.fetch_add(1, Ordering::Relaxed); // ~5 ns
// vs: Mutex::lock() → ~1000 ns under contention

Memory Ordering

CPUs and compilers reorder operations for performance. Atomic operations with memory ordering control this:

  • Relaxed: no ordering guarantees — just atomicity. Use for counters where stale reads are acceptable.
  • Acquire: all subsequent reads/writes happen after this load.
  • Release: all prior reads/writes happen before this store.
  • SeqCst: full sequential consistency — strongest, most expensive.

Keethu uses Relaxed ordering for the connection counter (conn_count) because seeing a slightly stale value in admin stats is fine. It uses stronger ordering for subscription state changes where consistency matters.

DashMap — Sharded Lock-Free HashMap

A Mutex<HashMap> serializes all reads and writes. DashMap shards the hash map into N buckets (default: 4× CPU count), each with its own RwLock. Operations on different keys rarely hit the same shard, dramatically reducing contention.

// Standard: all operations serialize through one lock
Mutex<HashMap<String, Vec<Sub>>>

// DashMap: 64 shards, each with its own RwLock
// "/sports" and "/finance" topics → different shards → no contention
DashMap<String, Vec<SubEntry>>

False Sharing Advanced

Two atomic counters that logically have nothing to do with each other can fight over the same CPU cache line, serializing what should be independent operations. The fix is padding.

// Without padding: conn_count and seq are likely on the same cache line
pub struct Engine {
    seq:        AtomicU64,  // 8 bytes
    conn_count: AtomicU64,  // 8 bytes — same 64-byte cache line!
}
// Core 0 writes seq, Core 1 writes conn_count → cache line ping-pong

// With padding: each counter on its own cache line
pub struct Counter {
    val: AtomicU64,
    _pad: [u8; 56], // 8 + 56 = 64 bytes = one full cache line
}

📖 FAQ — Performance Engineering

How do you measure p99 latency accurately?

Use an HDR (High Dynamic Range) histogram, which uses compressed logarithmic buckets to accurately represent latency distributions from 1µs to 1 hour in a single data structure with ~2% error. hdrhistogram is the Rust crate. Record a timestamp when a message enters Keethu and when it's delivered to the subscriber, compute the difference. Run for at least 60 seconds at target load before reading percentiles — the first seconds include JIT warm-up effects (Tokio's work-stealing scheduler needs to warm up).

What causes "coordinated omission" in benchmarks and why is it dangerous?

Coordinated omission: when your benchmark only sends the next request after the previous one completes, it never sees the queuing delay that real clients experience. If the server is slow, the benchmark naturally slows down — hiding the problem. Fix: use a load generator that sends requests at a fixed rate (e.g., with a timer), regardless of whether previous responses arrived. wrk2 and bombardier handle this correctly.

Why is jemalloc faster than the default glibc allocator?

glibc's malloc uses a global arena with a coarse lock. Under concurrent allocation from many threads, threads contend on this lock. jemalloc uses per-thread arenas (no sharing, no locks for common allocations) and size-class segregation (different pools for 8-byte, 16-byte, 32-byte objects, etc.) which reduces fragmentation. For Keethu, every incoming frame allocates a BytesMut; under high concurrency, jemalloc reduces allocation latency by 3-10×. Enable with: #[global_allocator] static ALLOC: jemallocator::Jemalloc = jemallocator::Jemalloc;

CHAPTER 5

Rust & Async — Why Rust for Keethu Intermediate

Rust makes one crucial promise: no garbage collector. GC languages (Java, Go, Python) periodically pause all threads to reclaim memory. A 50ms GC pause in a Java message broker means every subscriber waited 50ms — that's 5× the entire p99 latency budget. Rust's ownership system eliminates GC entirely by tracking at compile time exactly when memory can be freed.

Ownership in One Paragraph

Every value has exactly one owner. When the owner goes out of scope, the value is freed — deterministically, with no GC pause. You can borrow a value temporarily (references) but the compiler enforces that you can't hold a reference after the value is freed. This eliminates an entire class of bugs (use-after-free, dangling pointers) and eliminates GC.

let payload = Bytes::from("hello");
let a = payload.clone(); // O(1): clone increments ref count
let b = payload.clone(); // O(1): another ref
// payload, a, b all share the same buffer
// Memory freed when ALL of them go out of scope
// No GC. No pause. Deterministic.

Async / Await — State Machine Compilation Intermediate

When you write async fn, the Rust compiler transforms it into a state machine. Each .await becomes a state transition point. The state machine stores only what's needed across that suspension point.

// What you write:
async fn handle_client(stream: TcpStream) {
    let frame = stream.read_frame().await; // suspend here until data arrives
    process(frame);
    stream.write(response).await;         // suspend here until write completes
}

// What the compiler generates (simplified):
enum HandleClientState {
    ReadingFrame { stream: TcpStream },
    Processing  { stream: TcpStream, frame: Frame },
    Writing     { stream: TcpStream },
    Done,
}
// This enum is ~few hundred bytes — not 8MB like a thread stack

Tokio Internals — Work-Stealing Scheduler Advanced

Tokio is Keethu's async runtime. It manages:

  • Worker threads: one per CPU core, each with a local task queue
  • I/O reactor: a thread managing an epoll instance, waking tasks when their sockets are ready
  • Work-stealing: idle workers steal tasks from busy workers' queues
CPU Core 0 CPU Core 1 CPU Core 2 ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ Worker Thread│ │ Worker Thread│ │ Worker Thread│ │ Local Queue │◄──steal│ Local Queue │◄──steal│ Local Queue │ │ [t1,t2,t3] │ │ [t4,t5] │ │ [] ← idle │ └──────────────┘ └──────────────┘ └──┬───────────┘ │ steals task from core 1 I/O Reactor Thread │ ┌─────────────────┐ │ │ epoll_wait() │ │ │ "socket 42 ready"│ │ │ wake task t6 │──►│ └─────────────────┘

Work-stealing keeps all cores busy without a central scheduler lock. When core 2 is idle, it steals work from core 1's queue — a lock-free operation using a Chase-Lev deque. This gives Tokio near-linear throughput scaling with core count.

Zero-Copy Bytes in Rust Advanced

The bytes crate provides Bytes — a reference-counted, immutable byte buffer. Cloning a Bytes object is O(1): it just increments an atomic reference counter and copies the pointer and length. No data is copied.

// Incoming frame: allocate once
let payload: Bytes = src.split_to(pay_len).freeze(); // O(1)

// Fan out to 100K subscribers: 100K clones, each O(1)
for sub in subscribers {
    delivery.payload = payload.clone(); // just ref-count increment
}
// Total data copied: ZERO. Memory freed when last clone drops.

This is how Keethu achieves fan-out to 100K subscribers without copying the message body 100K times. The payload is allocated once on receipt and reference-counted to all subscribers.

DashMap — Concurrent HashMap Without Global Lock Advanced

Keethu's subscription registry maps topic names to subscriber lists. Under high concurrency (many publishes and subscribes simultaneously), a single Mutex<HashMap> becomes a bottleneck: all 32 CPU cores queue up waiting for the same lock.

DashMap shards the map into N independent buckets (N = 4 × CPU count by default = 128 on a 32-core machine). A topic hashes to exactly one shard. Operations on different topics hit different shards with independent RwLocks — full parallelism.

// Engine::subscribe() in keethu-server/src/engine.rs
// DashMap::entry() acquires only the shard lock for this topic
self.topics
    .entry(topic)      // locks shard N only
    .or_default()
    .push(SubEntry { conn_id, sub_id, tx });
// Other topics' shards are completely unaffected

📖 FAQ — Rust & Async

What is the difference between async/await and threads? When do you use each?

Threads are OS-managed, each with an ~8MB stack, preemptively scheduled. Use threads for CPU-bound work (computation) — they truly run in parallel. Async tasks are user-space state machines (~hundreds of bytes), cooperatively scheduled at await points. Use async for I/O-bound work (waiting for network, disk) — they're much cheaper than threads when waiting. Keethu's connection handler is async (I/O-bound: mostly waiting for client data). Keethu's future content filter (CPU-bound: SQL evaluation) may use Rayon thread pools.

What is "pinning" in Rust async and why does it exist?

An async state machine may hold self-references (a reference to data within itself). If the state machine is moved in memory, the self-references become invalid. Pin<T> is a type that prevents moving a value after pinning. You rarely interact with Pin directly — Tokio handles it internally. It comes up when implementing Future manually or using tokio::pin!(future) before select!.

Why use mpsc channels between the read task and write task?

Keethu's connection handler splits into two independent async tasks: a read task that processes incoming commands, and a write task that drains a channel and sends frames. Without this split, a slow write (blocked waiting for the socket to be writable) would stall processing incoming commands from the same client. The channel decouples them: the read task processes commands at full speed and sends responses to the channel; the write task drains the channel to the socket at whatever rate the socket allows. The bounded channel provides back-pressure: if the write task is behind, the channel fills up, causing send().await to suspend the read task — slowing down the pipeline naturally.

What does "Send + Sync" mean in Rust and why does Tokio require it?

Send means a type can be safely transferred between threads. Sync means a type can be safely accessed from multiple threads simultaneously (via shared references). Tokio's work-stealing scheduler may move a task from one worker thread to another — so everything in a Tokio task must be Send. Rc<T> (non-atomic ref count) is not Send — use Arc<T> instead. Cell<T> is not Sync — use AtomicT or Mutex<T>. The compiler enforces these at compile time — no runtime races possible.

Fan-Out Engine Architecture Pro

The publish path is the critical hot path. Every nanosecond counts.

pub fn publish(&self, topic: &str, payload: Bytes, publisher: &str) -> u64 {
    // Step 1: Atomic sequence number — no mutex, ~5 ns
    let seq = self.seq.fetch_add(1, Ordering::Relaxed);
    let ts  = chrono::Utc::now().timestamp_millis();

    // Step 2: DashMap shard lock for this topic only — ~20 ns
    if let Some(subs) = self.topics.get(topic) {
        // Step 3: Iterate subscribers — Vec = cache-friendly
        for sub in subs.iter() {
            // Step 4: Build delivery frame — Bytes clone is O(1)
            let frame = delivery_frame(&sub.sub_id, topic, &payload, seq, ts, publisher);
            // Step 5: try_send to per-connection channel — lock-free — ~10 ns
            if sub.tx.try_send(frame).is_err() {
                warn!("slow consumer dropped: conn {} sub {}", sub.conn_id, sub.sub_id);
            }
        }
    }
    seq // caller gets the sequence number
}

Per subscriber: ~35 ns. At 100K subscribers: 3.5ms for the fan-out loop — within the 10ms budget, with 6.5ms remaining for network jitter, TCP buffer flushes, and scheduling.

Phase 2 Optimization (Rayon parallel fan-out)

For topics with thousands of subscribers, Keethu will split the subscriber list across Rayon threads (one per CPU core) and fan-out in parallel, reducing 100K subscriber delivery from ~3.5ms to ~0.5ms on a 32-core machine.

Back-Pressure Architecture Pro

Back-pressure is the mechanism by which a slow consumer signals to producers to slow down. Without it, fast producers fill memory until the system crashes. Keethu implements back-pressure at every layer:

📖 FAQ — Keethu Architecture

Why use a binary protocol with JSON headers instead of pure binary or pure text?

Pure binary (protobuf, flatbuffers): fastest parse, but brittle — adding a field requires a schema change and recompile of all clients. Pure JSON text: flexible, but no framing (where does one message end?), verbose, slow to parse. Keethu's hybrid: 16-byte binary header provides length-prefixed framing with O(1) decode, while JSON headers provide schema flexibility — you can add new fields without breaking old clients. The payload is raw bytes — the application defines its own format, Keethu doesn't care.

Why does Keethu use "KEET" (0x4B454554) as the magic number?

The magic number's purpose is to detect protocol mismatches — if a client accidentally connects a random TCP client to Keethu's port, the first 4 bytes of their traffic almost certainly won't be 0x4B454554. If they don't match, Keethu closes the connection immediately with an InvalidMagic error rather than trying to parse garbage as a frame. The specific bytes encode the project name ("KEET" from Keethu) as ASCII, making wire captures easy to read: 00000000: 4b45 4554 01... KEET...

How does Keethu guarantee message ordering?

Within a single topic, Keethu assigns a monotonically increasing global sequence number (AtomicU64 fetch_add) to each published message. Subscribers receive the sequence field in each PUBLISH_DELIVERY frame. Ordering is guaranteed from a single publisher's perspective — messages are delivered in the order they were published. However, if two publishers publish to the same topic simultaneously, their messages are sequenced atomically (fetch_add is serialized at the CPU level), but the order between their messages depends on which publish() call wins the atomic increment first.

What is the difference between Keethu and Kafka?

Kafka is a distributed log — it persists every message to disk (with replication), supports replay from any offset, and trades latency (~5–100ms typical) for durability and horizontal scale across dozens of brokers. Keethu V1 is an in-memory pub/sub bus — it targets <10ms p99 on a single machine, supports live subscribers only (no durable replay in V1), and trades disk durability for speed. They solve different problems. Keethu's V2 SOW (State of the World) + txlog adds durable replay on a single machine — closer to AMPS, which is the inspiration.

Could we use io_uring instead of epoll for lower latency?

Yes, and it's on the roadmap. io_uring (Linux ≥5.1) submits I/O operations to a shared ring buffer between user space and kernel, avoiding most syscall overhead. For sequential workloads (write → read → write), io_uring chains can eliminate context switches entirely. Tokio has an experimental io_uring backend (tokio-uring). Expected gain: ~20–30% reduction in I/O overhead. The tradeoff is requiring Linux 5.10+ and giving up macOS/Windows compatibility for the server binary.

How does Keethu handle 100K connections without running out of file descriptors?

Each TCP connection consumes one file descriptor. Default Linux limit is 1,024 per process. Three changes are needed: (1) ulimit -n 1048576 in the shell before starting Keethu (or set in systemd unit via LimitNOFILE=1048576); (2) sysctl -w fs.file-max=2000000 for the system-wide limit; (3) ensure the Keethu process inherits the high limit. Keethu's systemd service file and Docker Compose config set these automatically via ulimits: nofile: 1048576.
